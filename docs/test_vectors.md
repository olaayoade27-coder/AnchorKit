# `compute_payload_hash` Test Vectors

Cross-language reference vectors for the `compute_payload_hash` function.
Off-chain clients (JavaScript, Python, etc.) must produce identical SHA-256 outputs
to be compatible with the on-chain Soroban contract.

## Algorithm

```
hash = SHA-256( subject_xdr || timestamp_be8 || data )
```

| Field | Encoding |
|---|---|
| `subject_xdr` | `Address.to_xdr()` — XDR-encoded `ScAddress` (see below) |
| `timestamp_be8` | 8-byte big-endian unsigned 64-bit integer |
| `data` | raw bytes, no length prefix |

### Subject XDR encoding (account address)

A Stellar G-address encodes as an XDR `ScAddress` of type `ScAddressTypeAccount`:

```
00 00 00 00          # ScAddress discriminant: ScAddressTypeAccount (0)
00 00 00 00          # PublicKey discriminant: PublicKeyTypeEd25519 (0)
<32 bytes>           # raw Ed25519 public key
```

Total: **40 bytes** for any account address.

The raw 32-byte Ed25519 key is decoded from the G-address strkey by:
1. Base32-decode the strkey (without padding)
2. Strip the 1-byte version prefix (`0x30`) and 2-byte checksum suffix
3. The remaining 32 bytes are the raw public key

### Off-chain pseudocode

```python
import hashlib, base64
from stellar_sdk import Keypair

def subject_xdr(strkey: str) -> bytes:
    raw_key = Keypair.from_public_key(strkey).raw_public_key()
    return b'\x00\x00\x00\x00' + b'\x00\x00\x00\x00' + raw_key  # 40 bytes

def compute_payload_hash(strkey: str, timestamp: int, data: bytes) -> bytes:
    msg = subject_xdr(strkey) + timestamp.to_bytes(8, 'big') + data
    return hashlib.sha256(msg).digest()
```

| Label | Strkey |

| `ADDR_A` | `GAAZI4TCR3TY5OJHCTJC2A4QSY6CJWJH5IAJTGKIN2ER7LBNVKOCCWN` |
| `ADDR_B` | `GCEZWKCA5VLDNRLN3RPRJMRZOX3Z6G5CHCGZXG5CPCJDGX2VOTZ5JXN` |

|---|---|

## Verification

The canonical expected hashes are generated by the Rust test suite in
`src/payload_hash_vectors_tests.rs`. Run:

```bash
cargo test payload_hash_vectors
```

Any off-chain implementation that produces the same SHA-256 outputs for these
inputs is compatible with the on-chain contract.
